# 計算知識論

## 前言

這是一套建立成功專案的流程（pipeline），而非學術研究的框架。我們借用研究中的概念來打造專案，但不追求學術上的嚴謹性或原創性。

專案的使命在於解決特定問題，只要能達成這一點，就算完成了它的價值。這套流程的重點在於務實，而不是理論上的完美。

## 流程

按照以下流程探索目標領域：

1. 問題空間
2. 抽象
3. 模型
4. 原型
5. 產品

不一定每個問題空間都會變成產品。實際上，很多問題空間只會停在模型或原型。這是探索的一環，不是失敗。因此，本流程的成功標準不在於產品，而在於是否深化了對問題空間的理解。

## 元框架

* 本流程並非傳統的軟體工程開發，而是一種知識探索的方法論。
* 若問題空間在探索過程中被判斷為缺乏實作價值，提早中止流程完全合理。這正是三段式實作設計的核心精神。
* 開發者甚至不用撰寫程式碼，只要能將問題空間映射為抽象並記錄理解，就已經完成了研究成果的一部分。
* 不過，我們仍鼓勵至少建立一個模型，作為初步理解問題空間的工具。之後若有需要，專案可以再度啟動，延伸至原型與產品階段。

此元框架不限於 DSL 或軟體開發，凡是能被形式化、可程式化的問題皆可套用。其核心精神在於「知識優先、產品次要」，因此任何能被抽象化並表達為模型的問題，都能透過此框架獲得研究價值。

## 程式碼的角色

在本流程中，程式碼不只是產品，而有多種認識論上的角色：

* **Formalization**：把抽象映射成可執行形式。
* **Verification**：檢查模型是否合理。
* **Exploration**：透過原型測試假設。
* **Communication**：程式碼作為知識共享的 artifact。

因此，程式碼在本流程中不僅是技術工具，更是知識的載體，承擔了從抽象到共享的完整認識論功能。

## 原則

* 程式碼是抽象的驗證工具。
* 研究成果可以停在任意層次。
* 產品是副產物，而非必然。
* 知識探索優先於交付。

## 對概念性驗證的反思

雖然產品是問題空間的副產品，但有價值的抽象應該朝向產品的方向發展，最終成為軟體基礎設施，而不是永遠停在模型或原型。

本流程的終極目標，是讓知識不僅停留在抽象或原型，而能轉化為持續可用的軟體基礎設施，成為研究與工程之間的橋樑。

## 問題空間

* 在目標領域中挑選具體問題，並將其縮放到合適的範圍，形成問題空間。
* 問題空間不是產品本身，而是待解決的命題；產品只是解決命題的工具。
* 在界定問題空間時，必須評估既有方案。這個過程能篩掉不值得投入的問題，節省寶貴的時間。
* 不要在沒有界定問題空間時就開始做產品。模糊的問題空間只會帶出沒有明確方向的產品。

問題不是「我想做一個程式語言」，而是「我想用程式語言來解決某個具體問題」，並檢視現有語言是否能解決。程式語言早已過剩，除非能帶來獨特特性來解決痛點，否則新的語言往往只會淪為 niche 專案。

## 抽象

* 抽象將問題空間映射到可操作的詞彙與結構。
* 建立抽象時，可以僅映射問題空間的一部分，並在需要時逐步精煉。
* 抽象不是演算法，也不是具體邏輯，而是模型的語言基礎。
* 部分抽象若對實作無幫助或不適合落地，可以在後續階段捨棄。
* 抽象的價值在於，它讓複雜的現象得以形式化，並成為模型與原型的起點。

例如，病歷包含個案資料、主訴、症狀、理學檢查、生化檢驗、影像檢查、診斷、治療等項目。這些資料並非全部嵌在同一份病歷中，而是分散在不同結構裡：個案資料可能獨立管理，檢驗結果則以時間序列存在。這樣的抽象在醫療資訊標準裡也常見，但本文不深入規範，只談抽象的角色。

## 模型

* 一開始先以自由文字記錄靈感。這時候不要過度在意正確性，把想法記錄下來比較重要。
* 等初步確認模型後，再根據情境以適當的工具建模。
* 建模時，可以寫死實字、只做最小範例，不需要寫出可用的程式。
* 不建議略過建模，直接做原型。建模時間更短，可及早篩除不合適的專案。

## 原型

* 原型的重點在於驗證概念可行性，因此選擇開發摩擦力小、撰寫速度快的技術棧。
* 開發原型時無須實作完整功能，僅需聚焦於核心功能。
* 經評估後，可直接將原型專案轉為成品而無需更換語言重寫，但代價是系統穩定度較低。
* 原型階段僅需專注於核心邏輯與命令列工具（CLI），切勿過早引入微服務等複雜架構。

> **範例**：若原型目標是開發一個標記語言（DSL），則只需確保 DSL 語法穩定，且具備轉換為 JSON 的核心路徑（Happy Path）即可視為完成任務。此階段不建議花費精力實作錯誤回報或語法檢查等非核心功能。

若直接採用成品語言開發原型，從概念到落地可能需要額外消耗三至五倍的開發時間；然而，這類高成本產出的半成品，有可能面臨不再繼續維護的風險。

## 產品

* 切勿在缺乏原型的情況下直接撰寫成品。沒有原型作為參考對象，會導致開發迭代速度過於緩慢。
* 由於已有原型作為實作基準，此階段甚至能透過 Vibe Coding 快速生成程式碼，再手動微調與優化。
* 在 Git 中建立一個新分支來撰寫成品專案，勿直接覆蓋或刪除既有的原型專案。
* 當成品專案開發成熟後，將原型專案移至獨立分支封存，並將成品專案的分支合併回 `main` 或 `master` 分支。

## 結語

同一個問題空間所映射出的抽象，會在模型、原型、產品三個階段以不同規模被重複實作。表面上看似浪費時間，但這正是流程的核心：**程式碼只是探索的產物，抽象模型才是知識的沉澱（Code as Epistemic Artifact）**。

透過反覆驗證，我們確認問題空間確實具有價值，並確保最終的產品不是偶然的成果，而是建立在穩固模型之上的落地實作。

## 願景

讓程式碼不僅是研究的附屬物，而是知識的持續載體，最終成為跨領域共享的基礎設施。
